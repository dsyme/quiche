# Formal Verification Research — quiche

> 🔬 *Generated by Lean Squad automated formal verification.*

## Overview

This document surveys the `quiche` codebase for formal verification (FV)
targets and documents the chosen approach.

**FV Tool**: Lean 4 with Mathlib  
**Primary focus**: Pure algorithmic components — encoding, data structure
invariants, numerical algorithms — where proofs are tractable and bugs have
security or correctness impact.

---

## Codebase Summary

`quiche` is Cloudflare's implementation of the QUIC transport protocol and
HTTP/3 in Rust (~218 Rust source files). The key modules from an FV
perspective are:

| Module | Language | Role |
|--------|----------|------|
| `octets/src/lib.rs` | Rust | Zero-copy byte buffer + QUIC varint codec |
| `quiche/src/ranges.rs` | Rust | Sorted non-overlapping range set |
| `quiche/src/recovery/rtt.rs` | Rust | RTT estimation (RFC 9002 §5) |
| `quiche/src/flowcontrol.rs` | Rust | QUIC flow control bookkeeping |
| `quiche/src/minmax.rs` | Rust | Windowed min/max filter (Kathleen Nichols' algorithm) |

## Target Survey

### Target 1: QUIC Variable-Length Integer Codec (`octets` crate)

**Location**: `octets/src/lib.rs`, functions `varint_len`, `varint_parse_len`,
`OctetsMut::put_varint`, `Octets::get_varint`

**Description**: QUIC (RFC 9000 §16) encodes integers as variable-length
2-bit-tag + value fields of 1, 2, 4, or 8 bytes. The codec has two pure
helper functions (`varint_len` and `varint_parse_len`) plus read/write
methods.

**Benefit**: The varint codec is used on every QUIC packet header and frame.
A bug here would silently corrupt all packet parsing. The round-trip property
(`decode(encode(v)) == v`) and the length-computation properties are highly
security-relevant.

**Specification size**: ~50–80 Lean lines to state the key properties.

**Proof tractability**: Excellent.
- `varint_len` is a 4-branch pure function over `u64` ranges — can be
  proved by `omega` or `decide` on each branch.
- `varint_parse_len` is a 4-case pure function on `u8 >> 6` — trivially
  decidable.
- Round-trip: `get_varint(put_varint(v)) == v` for `v ≤ MAX_VAR_INT` —
  requires modelling the buffer state, but the pure integer operations are
  provable by `omega`/`ring`.
- Consistency: `varint_parse_len(first_byte) == varint_len(v)` when
  `first_byte` is the first byte produced by `put_varint(v)`.

**Approximations needed**: The mutable buffer state (`OctetsMut`) must be
modelled as a pure byte sequence. The `unsafe` pointer arithmetic in the
`put_u`/`get_u` macros is abstracted away — we model only the integer
semantics, not the memory layout.

**Approach**: Functional Lean definitions of `varintLen` and `varintParseLen`,
then `decide`/`omega` proofs. Round-trip via equational reasoning.

**Priority**: ⭐⭐⭐ HIGHEST — pure, tractable, high security value.

### Target 2: Sorted Non-Overlapping Range Set (`ranges.rs`)

**Location**: `quiche/src/ranges.rs`

**Description**: A `RangeSet` maintains a sorted collection of
non-overlapping `Range<u64>` values, backed by either a `SmallVec` (for
≤4 ranges) or a `BTreeMap` (for larger sets). The key operations are
`insert` (which merges overlapping/adjacent ranges) and `remove_until`.

**Benefit**: The `RangeSet` is used to track ACK ranges in QUIC. An
incorrect implementation could allow duplicate ACK credit or miss packet
retransmissions — critical for reliability. The sortedness and
non-overlapping invariants are the key correctness criteria.

**Specification size**: ~100–150 Lean lines for types + invariant + 3–4 theorems.

**Proof tractability**: Medium.
- The invariant (sorted, non-overlapping) can be stated cleanly.
- Proving `insert` preserves the invariant requires case analysis on
  overlap detection — doable with `omega` + careful case splits.
- The dual Inline/BTree representation complicates things; we can model
  the abstract value (sorted list of ranges) ignoring representation.

**Approximations needed**: Abstract away the `SmallVec`/`BTreeMap` duality;
model as a sorted `List (Nat × Nat)`. Ignore capacity-eviction behaviour.

**Approach**: Define a `WF` well-formedness predicate (sorted, non-overlapping),
prove `insert` preserves it by induction on the list.

**Priority**: ⭐⭐ HIGH — data structure invariant, medium difficulty.

### Target 3: RTT Estimation (`recovery/rtt.rs`)

**Location**: `quiche/src/recovery/rtt.rs`, function `RttStats::update_rtt`

**Description**: Implements RFC 9002 §5 RTT estimation:
`smoothed_rtt = 7/8 * smoothed_rtt + 1/8 * adjusted_rtt`
`rttvar = 3/4 * rttvar + 1/4 * |smoothed_rtt - adjusted_rtt|`

**Benefit**: Incorrect RTT estimation leads to poor congestion control
decisions (too aggressive or too conservative). Properties include:
- After initialisation, `smoothed_rtt` is always positive.
- `smoothed_rtt` is a weighted average, so it is bounded between
  `min_rtt` and `max_rtt` after convergence.
- `rttvar` is always non-negative.

**Specification size**: ~80–120 Lean lines.

**Proof tractability**: Medium.
- The EWMA update rule is amenable to `linarith`/`omega` for
  integer arithmetic. Duration arithmetic needs care (nanoseconds).
- Strict positivity after initialisation: straightforward.
- Boundedness by min/max: requires inductive argument.

**Approximations needed**: Model `Duration` as `Nat` (nanoseconds).
Ignore `Instant` (time only matters for `Minmax` window, abstracted here).
Ignore `max_ack_delay` clamping in a first pass.

**Approach**: Functional model of `update_rtt` as a pure state transformer;
prove invariants by induction on number of updates.

**Priority**: ⭐⭐ HIGH — numerically subtle, RFC-specified correctness criteria.

### Target 4: Flow Control Bookkeeping (`flowcontrol.rs`)

**Location**: `quiche/src/flowcontrol.rs`

**Description**: Tracks bytes consumed vs. max_data limit. The
`should_update_max_data` predicate decides when to send a MAX_DATA frame.

**Benefit**: Flow control bugs can allow peers to send more data than
permitted (security) or prevent legitimate data (availability).
Key invariant: `consumed ≤ max_data` should hold at all times (modulo
pending updates).

**Specification size**: ~60–80 Lean lines.

**Proof tractability**: Good.
- The `should_update_max_data` logic is pure arithmetic, provable by
  `omega`.
- The `consumed ≤ max_data` invariant is simple to state and prove
  given the update functions.
- `update_max_data` monotonicity: `max_data` only increases.

**Approximations needed**: Ignore `Instant` (timing for autotuning window).
Model as pure arithmetic state.

**Approach**: Functional state model; `omega` proofs for arithmetic.

**Priority**: ⭐ MEDIUM — simpler, good warm-up target.

### Target 5: Windowed Min/Max Filter (`minmax.rs`)

**Location**: `quiche/src/minmax.rs`

**Description**: Kathleen Nichols' algorithm for tracking the minimum (or
maximum) of a data stream over a sliding time window. Used for `min_rtt`
tracking.

**Benefit**: An incorrect min-filter would cause the congestion controller
to underestimate or overestimate the baseline RTT, leading to poor bandwidth
utilisation. Key property: after `reset(t, v)`, the reported value is `v`.
After `running_min`, the reported value is ≤ all measurements in the window.

**Specification size**: ~80–100 Lean lines.

**Proof tractability**: Medium.
- The invariant "reported value is the minimum over the window" requires
  reasoning about time windows — harder to prove formally without a full
  trace model.
- Simpler subproperties (post-reset value, monotone decrease on new min)
  are tractable.

**Approximations needed**: Model `Instant` as `Nat`. Ignore the 3-sample
approximation detail (which is a known bounded approximation to the true
window min).

**Approach**: Prove simpler properties first (`reset` postcondition, new-min
path); leave window-correctness as a stated-but-unproved theorem.

**Priority**: ⭐ MEDIUM — interesting algorithm, partial proofs feasible.

## Tool Choice Rationale

**Lean 4 + Mathlib** is chosen because:

1. Rust's ownership/borrowing can be modelled functionally in Lean by
   extracting the pure computational content.
2. Mathlib provides rich automation (`omega`, `linarith`, `ring`, `simp`)
   well-suited to the arithmetic-heavy properties here.
3. The `decide` tactic can close many small decidable propositions
   automatically, given the bounded integer domains involved.
4. Lean 4 has good ergonomics for this style of protocol specification.

**Aeneas** (automatic Rust → Lean extraction) is worth attempting for
`varint_len` and `flowcontrol.rs` once initial hand-written specs are
established — it would give mechanically verified correspondence to the Rust.

## Web Research Notes

QUIC varint encoding is specified in RFC 9000 §16 and has well-understood
formal properties. The QUIC working group has no existing Lean/Coq
mechanisation known at time of writing. Mathlib has `Nat.log2` and bit
manipulation lemmas useful for the varint domain.

The Nichols min-filter has been analysed informally in BBR literature but
not formally verified in Lean/Coq to our knowledge.
