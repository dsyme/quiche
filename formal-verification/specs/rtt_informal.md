**RFC reference**: [RFC 9002 §5](https://www.rfc-editor.org/rfc/rfc9002#section-5)

## Purpose

`RttStats` implements the QUIC RTT estimation algorithm from RFC 9002 §5. It
maintains three key statistics about round-trip time:

- **`smoothed_rtt` (SRTT)**: exponentially-weighted moving average of RTT samples
- **`rttvar` (RTTVAR)**: mean deviation of RTT — used to compute the
  retransmission timeout (PTO)
- **`min_rtt`**: the minimum RTT observed over a configurable time window
  (used to detect implausible acknowledgement delays)

These statistics are updated each time an ACK frame is processed, using the
measured latest RTT sample.

---

## Preconditions

### `new(initial_rtt, max_ack_delay)`
- `initial_rtt > 0` (meaningful initial estimate, e.g. the initial timeout)
- `max_ack_delay ≥ 0`

### `update_rtt(latest_rtt, ack_delay, now, handshake_confirmed)`
- `latest_rtt > 0` (RTT sample must be positive; zero would indicate a bug)
- `ack_delay ≥ 0`
- `now` is a monotonically non-decreasing `Instant`

---

## Postconditions

### `new(initial_rtt, max_ack_delay)` → `RttStats`
- `smoothed_rtt = initial_rtt`
- `rttvar = initial_rtt / 2`
- `min_rtt = initial_rtt` (windowed min, reset to initial)
- `has_first_rtt_sample = false`
- `latest_rtt = 0` (no sample yet)

### `update_rtt(latest_rtt, ...)` — first call (when `!has_first_rtt_sample`)
- `smoothed_rtt = latest_rtt`
- `rttvar = latest_rtt / 2`
- `min_rtt = latest_rtt`
- `has_first_rtt_sample = true`

### `update_rtt(latest_rtt, ack_delay, ...)` — subsequent calls
Let `adjusted_rtt` be the **ack-delay adjusted** RTT (see below).

- `smoothed_rtt = smoothed_rtt_prev * 7/8 + adjusted_rtt / 8`
  (EWMA with weight 1/8 on new sample — RFC 9002 §5.3)
- `rttvar = rttvar_prev * 3/4 + |smoothed_rtt_prev - adjusted_rtt| / 4`
  (mean deviation EWMA — RFC 9002 §5.3)
- `min_rtt = min(min_rtt_prev, latest_rtt)` (simplified; actual impl uses a
  windowed min over `RTT_WINDOW = 300s`)
- `has_first_rtt_sample = true`

---

## Acknowledgement-Delay Adjustment (RFC 9002 §5.4)

**`adjusted_rtt`** is computed as follows:

1. If `handshake_confirmed`, clamp `ack_delay = min(ack_delay, max_ack_delay)`
2. If `latest_rtt ≥ min_rtt + ack_delay` (**plausible** delay):
   `adjusted_rtt = latest_rtt - ack_delay`
3. Otherwise (**implausible** delay — peer reported more delay than physically
   possible given `min_rtt`):
   `adjusted_rtt = latest_rtt` (ignore the reported delay)

**Rationale**: the minimum RTT represents the propagation delay (no queuing,
no ack delay). If `latest_rtt - ack_delay < min_rtt`, the reported ack delay
is inconsistent with the measured minimum, so we ignore it.

---

## Invariants

**I1 — SRTT is always positive** (given `initial_rtt > 0` and `latest_rtt > 0`):
After any update, `smoothed_rtt > 0`.

**I2 — SRTT is bounded by recent samples**:
`smoothed_rtt * 7/8 + adjusted_rtt / 8 ≤ max(smoothed_rtt, adjusted_rtt)`
(EWMA always lies within the convex hull of old and new values)

**I3 — `adjusted_rtt ≤ latest_rtt`** (delay adjustment only reduces RTT):
`adjusted_rtt ≤ latest_rtt`

**I4 — min_rtt is non-increasing**:
After each update, `min_rtt_new ≤ min_rtt_prev`

**I5 — `has_first_rtt_sample` is monotonically set** (never goes false → true → false):
Once `has_first_rtt_sample = true`, it remains true forever.

**I6 — rttvar ≥ 0** (trivially holds in non-negative arithmetic):
`rttvar ≥ 0` always.

---

## Edge Cases

- **Zero ack_delay**: `adjusted_rtt = latest_rtt` (ack_delay ≥ 0 and
  `latest_rtt ≥ min_rtt + 0 = min_rtt` iff `latest_rtt ≥ min_rtt`; if even
  this fails, it's still safe to use `latest_rtt`).
- **ack_delay > latest_rtt**: implausible, `adjusted_rtt = latest_rtt`.
- **Very first sample = initial_rtt**: smoothed_rtt unchanged, rttvar = 0/2 = 0.
- **Stable RTT (all samples equal)**: SRTT converges to the sample value;
  RTTVAR shrinks toward 0.
- **Integer division truncation**: all arithmetic uses integer (floor) division.
  `rttvar * 3 / 4` may lose up to 3 nanoseconds per update.

## Examples

### Initialization (initial_rtt = 100ms = 100_000_000 ns)
```
smoothed_rtt = 100_000_000
rttvar       = 50_000_000  (= 100_000_000 / 2)
min_rtt      = 100_000_000
```

### First sample (latest_rtt = 120ms, no ack_delay)
```
adjusted_rtt  = 120_000_000  (no adjustment since has_first = false path)
smoothed_rtt  = 120_000_000
rttvar        = 60_000_000
```

### Second sample (latest=100ms, min=80ms, ack_delay=10ms)
```
adjusted_rtt  = 100_000_000 - 10_000_000 = 90_000_000
              (plausible: 100ms ≥ 80ms + 10ms)
smoothed_rtt  = prev * 7/8 + 90_000_000/8 = 87_500_000 + 11_250_000 = 98_750_000
                (assuming prev = 100ms)
rttvar        = prev * 3/4 + |100_000_000 - 90_000_000|/4
              = 20_000_000 * 3/4 + 10_000_000/4
              = 15_000_000 + 2_500_000 = 17_500_000
```

## Inferred Intent

The EWMA constants `7/8` and `3/4` (and `1/8`, `1/4` for the new sample weight)
are fixed by RFC 9002. They represent a relatively slow-moving average (8-sample
time constant) that avoids overreacting to individual RTT spikes.

The `abs_diff` (not signed difference) in the rttvar formula means RTTVAR tracks
absolute deviation — it grows both when RTT increases and when it decreases.

The `min_rtt` tracking serves as a "sanity floor": even if intermediate ACKs
have high ack_delay, the minimum tells us the inherent propagation delay.

## Open Questions

1. **Overflow risk**: `smoothed_rtt.as_nanos().abs_diff(adjusted_rtt.as_nanos()) as u64 / 4`
   casts from u128 to u64. If `abs_diff > u64::MAX`, this truncates silently.
   Is this a realistic concern? (Would require RTT > ~580 years, so probably not.)

2. **`max_rtt` field**: the struct tracks `max_rtt` but there are no public accessors.
   Is it used internally for any invariant?

3. **Windowed `min_rtt`**: the actual Rust code uses `Minmax::running_min(RTT_WINDOW, ...)`
   which evicts old samples after 300 seconds. The formal model uses `Nat.min` (simple
   minimum). Are there security properties that rely on the windowed eviction?

4. **`loss_delay` time_threshold**: the `time_thresh: f64` parameter is `9/8` by default
   (RFC 9002 §6.1.2). The Lean model uses a rational approximation. Acceptable?
