**Key struct**: `DatagramQueue<F: BufFactory>` — a bounded FIFO queue of DATAGRAM frames.

---

## Purpose

`DatagramQueue` manages an ordered collection of QUIC DATAGRAM frames awaiting
transmission or delivery. Its two core responsibilities are:

1. **Capacity enforcement**: the queue holds at most `queue_max_len` datagrams;
   attempts to push beyond this limit are rejected.
2. **Byte-size accounting**: `queue_bytes_size` always equals the sum of the
   lengths of all datagrams currently in the queue, so callers can ask "how
   many bytes are pending?" without iterating.

The queue is a FIFO: `push` enqueues at the back; `pop` dequeues from the front.

---

## State

| Field | Type | Description |
|-------|------|-------------|
| `queue` | `Option<VecDeque<F::DgramBuf>>` | The underlying deque; `None` means empty (lazy init) |
| `queue_max_len` | `usize` | Maximum number of datagrams that may be held |
| `queue_bytes_size` | `usize` | Running sum of `buf.len()` for every datagram in the queue |

### Key derived values

- `len()` = number of datagrams in the queue (0 if `queue` is `None`)
- `is_empty()` iff `len() == 0`
- `is_full()` iff `len() == queue_max_len`
- `byte_size()` = `queue_bytes_size`

---

## Invariants

**I1 (Capacity bound)**: At all times, `len() ≤ queue_max_len`.

**I2 (Byte-size consistency)**: At all times,
`byte_size() = Σ { d.len() | d ∈ queue }`.

**I3 (Non-negative sizes)**: `byte_size() ≥ 0` and every individual datagram
length ≥ 0 (trivially true since `usize` is unsigned, but worth stating
explicitly for the Lean model).

**I4 (Empty consistency)**: `is_empty()` iff `byte_size() == 0` when every
datagram in the queue has length > 0. (Note: if a datagram with `len = 0` were
ever pushed, `is_empty()` and `byte_size() == 0` would diverge; this edge case
is not explicitly guarded by the code.)

## Preconditions and Postconditions

### `new(queue_max_len)`
- **Pre**: none
- **Post**: `len() == 0`, `byte_size() == 0`, `queue_max_len` is set to the given value

### `push(data)`
- **Pre**: none (caller must tolerate an `Err(Error::Done)` when full)
- **Post (success)**: `len_after = len_before + 1`, `byte_size_after = byte_size_before + data.len()`, the new datagram is at the back of the queue
- **Post (failure — queue full)**: returns `Err(Error::Done)`; queue is unchanged
- **Error condition**: `is_full()` → returns `Err(Error::Done)`

### `pop()`
- **Pre**: none
- **Post (non-empty)**: returns `Some(front)`, `len_after = len_before - 1`, `byte_size_after = byte_size_before - front.len()`, FIFO order preserved (the returned element was the front)
- **Post (empty)**: returns `None`; queue is unchanged

### `peek_front_len()`
- **Pre**: none
- **Post**: returns `Some(len)` where `len` is the length of the front datagram, or `None` if empty
- **Non-mutating**: the queue is unchanged

### `peek_front_bytes(buf, len)`
- **Pre**: `buf.len() ≥ min(len, front_datagram.len())`
- **Post (non-empty)**: copies `min(len, front_datagram.len())` bytes into `buf`, returns `Ok(bytes_copied)` — the queue is unchanged
- **Post (empty)**: returns `Err(Error::Done)`
- **Error condition (buffer too short)**: if `buf.len() < min(len, front.len())` → returns `Err(Error::BufferTooShort)`

### `has_pending()`
- **Post**: equivalent to `!is_empty()`

### `is_full()`
- **Post**: `len() == queue_max_len`

### `is_empty()`
- **Post**: `len() == 0`

### `len()`
- **Post**: number of datagrams in the queue; 0 when `queue` is `None`

### `byte_size()`
- **Post**: `queue_bytes_size` (consistent with I2)

### `purge(f)`
- **Pre**: `f` is a predicate over byte slices
- **Post**: all datagrams `d` with `f(d) == true` are removed; `byte_size` is
  recomputed as the sum of lengths of the remaining datagrams; relative order
  of surviving datagrams is preserved (stable)
- **Note**: `byte_size` is fully recomputed (not incrementally updated), which
  is correct but potentially slower than incremental maintenance

## Edge Cases

1. **Empty queue, pop**: returns `None`, queue unchanged.
2. **Full queue, push**: returns `Err(Error::Done)`, queue unchanged.
3. **Single element**: after `push` then `pop`, queue is empty and `byte_size == 0`.
4. **Zero-length datagram**: `push` accepts it; `byte_size` unchanged; `len` increases; `is_empty()` and `byte_size() == 0` diverge.
5. **max_len = 0**: `is_full()` immediately, every `push` fails.
6. **purge removes all**: `len() == 0`, `byte_size() == 0`.
7. **purge removes none**: queue unchanged.
8. **byte_size saturating_sub in pop**: uses `saturating_sub` so can never underflow, but under I2 the subtraction should never underflow anyway.

## Examples

```
let mut q = DatagramQueue::new(3);
assert q.len() == 0, q.is_empty() == true

q.push([1,2,3])   → Ok(())
q.push([4,5])     → Ok(())
q.push([6])       → Ok(())
q.push([7,8,9])   → Err(Done)  -- full

q.len() == 3, q.byte_size() == 6, q.is_full() == true

q.pop() → Some([1,2,3])
q.len() == 2, q.byte_size() == 3

q.peek_front_len() → Some(2)
```

## Inferred Intent

The queue exists to buffer datagrams for asynchronous dispatch. The capacity
limit prevents unbounded memory use. The byte-size counter allows the QUIC
connection to compute "how much datagram data is queued?" without traversing
the deque. The lazy `Option<VecDeque>` avoids heap allocation when no datagrams
are used.

## Open Questions

1. **Zero-length datagrams**: should `push` reject them? The code does not,
   but a zero-length QUIC DATAGRAM frame is meaningless. Should I2 be
   strengthened to require all lengths > 0?
2. **`purge` recompute vs. incremental**: the implementation recomputes
   `byte_size` from scratch on every `purge`. Is this intentional (simplicity)?
   Could it become a performance issue for large queues?
3. **`queue_max_len = 0`**: is this a valid configuration? No guard prevents it;
   the queue is immediately full.
