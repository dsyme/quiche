# Informal Specification: NewReno Congestion Window

🔬 *Generated by Lean Squad automation.*

**Rust source**: `quiche/src/recovery/congestion/reno.rs`
**Supporting module**: `quiche/src/recovery/congestion/mod.rs`
**Key constants** (`quiche/src/recovery/mod.rs`):
- `LOSS_REDUCTION_FACTOR = 0.5`
- `MINIMUM_WINDOW_PACKETS = 2`

---

## Purpose

The NewReno (RFC 6582) congestion-control algorithm governs how many bytes
a QUIC endpoint is permitted to send in response to acknowledgements (ACKs)
and losses.  The algorithm tracks two numbers:

- **`congestion_window` (cwnd)**: the current sending limit in bytes.
- **`ssthresh`**: the slow-start threshold; when `cwnd < ssthresh` the
  endpoint is in *slow start* (exponential growth); when `cwnd ≥ ssthresh`
  it is in *congestion avoidance* (linear growth).

The goal is for the window to grow towards the bottleneck bandwidth and then
react proportionally to congestion signals (loss events).

---

## State

```
Congestion {
    congestion_window: usize,          -- current cwnd in bytes
    ssthresh: usize,                   -- default: usize::MAX (initial slow start)
    bytes_acked_sl: usize,             -- ACKed bytes counter for slow start
    bytes_acked_ca: usize,             -- ACKed bytes counter for congestion avoidance
    congestion_recovery_start_time: Option<Instant>,  -- when recovery began
    max_datagram_size: usize,          -- MSS (maximum segment size) in bytes
    initial_congestion_window_packets: usize,  -- initial cwnd in packets
    app_limited: bool,                 -- sender is application-limited
    hystart: Hystart,                  -- HyStart++ state (optional)
}
```

### Representation invariants

1. `congestion_window ≥ max_datagram_size * MINIMUM_WINDOW_PACKETS`
   — i.e. `cwnd ≥ 2 * mss` at all times after the first congestion event.
2. `congestion_window > 0` at all times (initial value is at least 1 packet).
3. `bytes_acked_ca < congestion_window` between `on_packet_acked` calls
   (the counter is reduced by `congestion_window` when it reaches that value).

## Operations

### `on_packet_acked(packet, now, rtt_stats)` — single-packet ACK callback

**Called by**: `on_packets_acked`, once per ACKed packet.

**Preconditions**:
- `packet.time_sent` is a valid past instant.
- `packet.size > 0`.

**Guard conditions** (no change if either holds):
- `in_congestion_recovery(packet.time_sent)` — packet was sent during
  an ongoing recovery period; window does not grow until recovery ends.
- `app_limited` — the sender is not filling the pipe; window does not grow.

**Slow-start branch** (`congestion_window < ssthresh`):

- Accumulate `bytes_acked_sl += packet.size`.
- If HyStart++ CSS is active (`hystart.in_css()`):
  - `congestion_window += hystart.css_cwnd_inc(mss)` (limited growth).
- Otherwise:
  - `congestion_window += mss` (one MSS per ACKed packet = exponential growth).
- After the update, call `hystart.on_packet_acked(packet, latest_rtt, now)`.
  If this returns `true` (CSS round ended), exit slow start:
  `ssthresh = congestion_window`.

**Congestion-avoidance branch** (`congestion_window ≥ ssthresh`):

- `bytes_acked_ca += packet.size`.
- When `bytes_acked_ca ≥ congestion_window`:
  - `bytes_acked_ca -= congestion_window`
  - `congestion_window += mss`
  — *one MSS increase per congestion-window's worth of ACKed bytes (AIMD)*.

**Postconditions**:
- If slow-start and not guarded: `congestion_window' ≥ congestion_window`.
- If congestion avoidance and not guarded: `congestion_window'` increases by
  at most `mss` per RTT worth of data.

---

### `congestion_event(largest_lost_pkt, now)` — loss detected

**Preconditions**:
- `largest_lost_pkt.time_sent` is the time the lost packet was sent.

**Guard**: if `in_congestion_recovery(time_sent)` then this packet was already
accounted for in a prior recovery period — **no action**.

**If a new congestion event**:

1. `congestion_recovery_start_time = Some(now)` — mark entry into recovery.
2. `congestion_window = max(congestion_window * LOSS_REDUCTION_FACTOR,
                             mss * MINIMUM_WINDOW_PACKETS)`
   — halve the window, but never go below the minimum.
3. `bytes_acked_ca = congestion_window * LOSS_REDUCTION_FACTOR`
   — reset the CA counter proportionally.
4. `ssthresh = congestion_window` — enter congestion avoidance.
5. Notify HyStart++ if in CSS.

**Postconditions**:
- `congestion_window' ≤ congestion_window / 2 + 1`
  (flooring due to integer arithmetic).
- `congestion_window' ≥ mss * MINIMUM_WINDOW_PACKETS` — floor invariant holds.
- `ssthresh' = congestion_window'` — mode is now congestion avoidance.

---

### `in_congestion_recovery(sent_time) → bool`

Returns `true` iff `sent_time ≤ congestion_recovery_start_time`.

**Key property**: once recovery begins at time `t`, only packets sent **after**
`t` can trigger further window reductions.  This prevents multiple halvings
for losses within the same RTT (the "single reduction per RTT" rule).

---

## Invariants and Key Properties

### Property 1 — Window floor

> After any congestion event, `congestion_window ≥ mss * 2`.

Source: `max(… , mss * MINIMUM_WINDOW_PACKETS)` in `congestion_event`.

### Property 2 — Window monotone in slow start (guarded)

> In slow start, if not in recovery and not app-limited, every ACK increases
> `congestion_window` by at least `hystart.css_cwnd_inc(mss) ≥ 1`.

### Property 3 — AIMD linear growth in congestion avoidance

> In congestion avoidance, `congestion_window` increases by exactly `mss` for
> every `congestion_window` bytes ACKed (one MSS per RTT on a full pipe).

### Property 4 — Single halving per loss epoch

> `congestion_event` only modifies `congestion_window` for packets sent
> **after** the start of the current recovery period.  Multiple losses in the
> same epoch do not compound reductions.

### Property 5 — Recovery tracking monotonicity

> `congestion_recovery_start_time` is set to `now` on each new congestion event
> and never decreases.  Therefore `in_congestion_recovery` is monotone with
> respect to time: once a `sent_time` stops being "in recovery", it never
> becomes "in recovery" again.

## Edge Cases

| Scenario | Expected behaviour |
|----------|-------------------|
| `congestion_window = mss * MINIMUM_WINDOW_PACKETS` on loss | Window stays at `mss * 2` (max() floor) |
| `packet.size = 0` | ACK callback is no-op (0 added to counters); window unchanged |
| Multiple losses in one epoch | Only the first triggers a halving; subsequent losses guarded by `in_congestion_recovery` |
| `app_limited = true` | No growth on ACK; avoids inflating cwnd when pipe is not full |
| Initial state (`ssthresh = usize::MAX`) | Always in slow start until first loss; window grows by 1 MSS per ACKed packet |
| `LOSS_REDUCTION_FACTOR * cwnd < mss * MINIMUM_WINDOW_PACKETS` | Window is clamped to `mss * MINIMUM_WINDOW_PACKETS` |

## Examples

### Example 1 — Slow-start growth

- Initial: `cwnd = 10 * mss`, `ssthresh = usize::MAX`, not in recovery.
- ACK arrives for a non-recovery, non-app-limited packet of `size = mss`.
- Expected: `cwnd = 11 * mss`.

### Example 2 — Congestion event

- Initial: `cwnd = 20 * mss`, `ssthresh = usize::MAX`.
- Loss event for a packet sent after the start of any prior recovery.
- Expected:
  - `cwnd = max(10 * mss, 2 * mss) = 10 * mss`.
  - `ssthresh = 10 * mss`.
  - `in_congestion_recovery(now) = true`.

### Example 3 — Congestion avoidance

- Initial: `cwnd = 10 * mss`, `ssthresh = 10 * mss`, `bytes_acked_ca = 0`.
- One RTT's worth of ACKs: `10 * mss` bytes ACKed one packet at a time.
- After `10 * mss` bytes: `bytes_acked_ca` has reached `cwnd`, so
  `cwnd = 11 * mss`, `bytes_acked_ca = 0`.

### Example 4 — ACK for packet sent during recovery (guarded)

- `congestion_recovery_start_time = Some(T)`.
- ACK arrives for packet sent at time `T - 1ms` (before recovery).
- Expected: **no change** to `cwnd`.

## Inferred Intent

The NewReno implementation directly follows RFC 6582.  The codebase also
supports CUBIC (`cubic.rs`) and uses a shared `Congestion` struct managed via
the `CongestionControlOps` vtable.  The NewReno-specific logic is concentrated
in three functions: `on_packet_acked`, `congestion_event`, and `on_packet_sent`
(the last being a no-op for Reno).

The `bytes_acked_ca` counter is a standard RFC 5681 trick to implement
"increase by 1 MSS per RTT" without tracking RTTs explicitly.  Its value at
the start of congestion avoidance is set to `cwnd * 0.5` to make growth start
at half-speed immediately after a congestion event (implementation detail, not
explicitly required by the RFC but commonly done).

## Open Questions for Maintainers

1. **`bytes_acked_ca` initialisation after congestion event**: the code sets
   `bytes_acked_ca = congestion_window * LOSS_REDUCTION_FACTOR`.  This means
   the CA counter starts at `cwnd/2` (i.e. half of the new `cwnd`), so only
   half a window of ACKs is needed before the first CA increment.  Is this
   intentional?  RFC 6582 does not specify initial counter values.

2. **Integer floor in `congestion_event`**: the window halving uses
   `(r.congestion_window as f64 * LOSS_REDUCTION_FACTOR) as usize`.  For odd
   `congestion_window` values this floors the result.  Is this floor behaviour
   required by the RFC, or should it round up?

3. **HyStart++ CSS interaction**: `css_cwnd_inc` controls the growth rate in
   conservative slow start.  If `css_cwnd_inc` returns 0 (edge case), `cwnd`
   never grows in CSS.  Should there be a minimum of 1 byte?

4. **`bytes_acked_sl` is never reset**: `bytes_acked_sl` accumulates across the
   entire connection lifetime and is never decremented.  Is it used elsewhere
   or is it a dead-weight counter?
