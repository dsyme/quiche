## Purpose

QUIC varints (RFC 9000 ┬¦16) are self-delimiting: the top 2 bits of the first
byte encode the total byte length of the integer. This spec targets the **tag
subsystem** ŌĆö the structural properties of those 2-bit tags ŌĆö independently of
the full round-trip proof (T23) and the varint value codec (T1).

Three functions collaborate to form the tag subsystem:

| Function | Role |
|----------|------|
| `varint_len(v)` | Chooses byte length from value range |
| `varint_parse_len(first)` | Reads byte length from first byte |
| `put_varint_with_len` (tag placement) | OR-s the 2-bit tag into first byte |

The key structural properties are:

1. **Range completeness**: `varint_parse_len` handles every possible first byte
   (`u8` range 0ŌĆō255) and returns exactly one of {1, 2, 4, 8}.
2. **Range bijectivity**: the four length classes map to four disjoint, complete
   intervals of the first byte ([0,63], [64,127], [128,191], [192,255]).
3. **Tag consistency**: `put_varint(v)` writes a first byte `b` such that
   `varint_parse_len(b) = varint_len(v)`.
4. **Tag injectivity**: different length classes produce first bytes in different,
   non-overlapping ranges ŌĆö a receiver can unambiguously determine the length.
5. **Value-range equivalence**: `varint_len(v) = n Ōåö v` is in the corresponding
   range. Equivalently, the four value ranges partition `[0, MAX_VAR_INT]`.
6. **Tag-value non-interference**: the 2-bit tag does not overlap with the value
   bits (the OR in `put_varint_with_len` is equivalent to addition).

These properties are more fine-grained than what T1/T23 prove. They are useful
for reasoning about packet parsing: a QUIC parser that reads only the first byte
can determine exactly how many more bytes to buffer before decoding.

### ┬¦3 ŌĆö Tag-bit placement: non-overlap proof

The Lean model encodes the tag by **addition** (not bitwise OR) because the
tag constant and the value bits occupy non-overlapping bit positions:

| Length | Value fits in | Tag constant | Non-overlap proof |
|--------|--------------|--------------|-------------------|
| 1 byte | 6 bits (v Ōēż 63) | 0 | trivial (tag = 0) |
| 2 bytes | 14 bits (v Ōēż 16383) | 0x4000 = 2^14 | v < 2^14, tag Ōēź 2^14 |
| 4 bytes | 30 bits (v Ōēż 2^30-1) | 0x80000000 = 2^31 | v < 2^30 < 2^31 |
| 8 bytes | 62 bits (v Ōēż 2^62-1) | 0xC000000000000000 | v < 2^62, tag uses bits 62ŌĆō63 |

Formally, for the 2-byte case: `v Ōēż 16383 ŌåÆ v + 0x4000 < 0x8000`, which
means the tag bit 14 is set and bits 15+ are clear ŌĆö the OR is addition.

**Lean propositions**:
```lean

    v + 2147483648 < 4294967296

    v + 13835058055282163712 < 18446744073709551616
```

## Invariants

- **Tag bits in MSB only**: the 2-bit tag is always in bits 6ŌĆō7 of the first
  byte. The value bits occupy bits 0ŌĆō5 (1-byte), 0ŌĆō13 (2-byte), 0ŌĆō29 (4-byte),
  or 0ŌĆō61 (8-byte). They never overlap.
- **Monotonicity preserved**: the non-overlap property implies that encoding a
  larger value with a longer tag still preserves value ordering within each class.

## Edge Cases

1. **`first = 63` (top of 1-byte range)**: `varint_parse_len(63) = 1` since
   `63 / 64 = 0`.
2. **`first = 64` (bottom of 2-byte range)**: `varint_parse_len(64) = 2` since
   `64 / 64 = 1`.
3. **`first = 191` (top of 4-byte range)**: `191 / 64 = 2`, so length = 4.
4. **`first = 192` (bottom of 8-byte range)**: `192 / 64 = 3`, so length = 8.
5. **`first = 255`**: `255 / 64 = 3`, so length = 8. This is the all-ones byte.
6. **Non-minimal encoding** (`put_varint_with_len(37, 2)`): first byte is
   `(37 + 16384) / 256 = 64`, which is in [64,127] ŌåÆ `varint_parse_len = 2`.
   Consistent with the requested length, not with `varint_len(37) = 1`.

## Inferred Intent

The 2-bit tag is a compact framing mechanism that makes QUIC varints
**self-delimiting** at zero overhead ŌĆö no length prefix is needed because the
length information is embedded in the first byte itself. The four byte ranges
([0,63], [64,127], [128,191], [192,255]) are chosen because they correspond to
the 4 values of 2-bit integers (0, 1, 2, 3) shifted to the MSBs.

The non-overlap between tag bits and value bits (┬¦3) is the key soundness
argument: the Lean model that replaces bitwise OR with addition is correct
precisely because this overlap is impossible given the range guards.
