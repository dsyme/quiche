## Purpose

The QUIC protocol (RFC 9000 §16) uses a compact variable-length integer
encoding that represents values in 1, 2, 4, or 8 bytes. The encoding
uses the 2 most significant bits of the first byte to indicate the total
byte length:

| MSBs (`first >> 6`) | Length | Max value |
|---------------------|--------|-----------|
| `00` | 1 byte | 63 (`0x3F`) |
| `01` | 2 bytes | 16383 (`0x3FFF`) |
| `10` | 4 bytes | 1 073 741 823 (`0x3FFFFFFF`) |
| `11` | 8 bytes | 4 611 686 018 427 387 903 (`MAX_VAR_INT`) |

The codec consists of four functions:

- `varint_len(v: u64) -> usize`: given a value, returns the minimum
  number of bytes required to encode it.
- `varint_parse_len(first: u8) -> usize`: given the first byte of an
  encoded varint, returns the total encoded length.
- `OctetsMut::put_varint(v: u64)`: encodes `v` into the buffer and
  advances the write offset by `varint_len(v)` bytes.
- `Octets::get_varint() -> u64`: decodes a varint from the buffer and
  advances the read offset by the encoded length.

## Preconditions

### `varint_len(v)`
- `v ≤ MAX_VAR_INT` (= 4 611 686 018 427 387 903 = 2^62 − 1).
- If `v > MAX_VAR_INT`, the function calls `unreachable!()` — this is a
  programmer error and is **not** a recoverable error.

### `varint_parse_len(first)`
- `first` is the first byte of a well-formed QUIC varint encoding.
- No precondition beyond this — all 256 values of `u8` are valid (the
  top 2 bits select the length unambiguously).

### `put_varint(v)`
- The buffer has at least `varint_len(v)` bytes of remaining capacity.
- `v ≤ MAX_VAR_INT`.
- If the buffer is too short, returns `Err(BufferTooShortError)` and
  does **not** advance the offset or modify the buffer.
- If `v > MAX_VAR_INT`, panics via `put_varint_with_len` with the
  message "value is too large for varint".

### `get_varint()`
- The buffer has at least `varint_parse_len(first_byte)` bytes of
  remaining capacity (where `first_byte` is the byte at the current
  offset).
- If there are insufficient bytes, returns `Err(BufferTooShortError)`.

## Postconditions

### `varint_len(v)` → `n`
- `n ∈ {1, 2, 4, 8}`.
- `n = 1` iff `v ≤ 63`.
- `n = 2` iff `64 ≤ v ≤ 16383`.
- `n = 4` iff `16384 ≤ v ≤ 1 073 741 823`.
- `n = 8` iff `1 073 741 824 ≤ v ≤ MAX_VAR_INT`.
- `varint_len(v) ≤ 8` for all valid `v`.
- `varint_len` is monotone: `a ≤ b → varint_len(a) ≤ varint_len(b)`.

### `varint_parse_len(first)` → `n`
- `n ∈ {1, 2, 4, 8}`.
- `n = 1` iff `first >> 6 = 0` (i.e., `first ≤ 0x3F`).
- `n = 2` iff `first >> 6 = 1` (i.e., `0x40 ≤ first ≤ 0x7F`).
- `n = 4` iff `first >> 6 = 2` (i.e., `0x80 ≤ first ≤ 0xBF`).
- `n = 8` iff `first >> 6 = 3` (i.e., `0xC0 ≤ first ≤ 0xFF`).

### `put_varint(v)` (success case)
- The buffer offset advances by exactly `varint_len(v)` bytes.
- The bytes written encode `v` per RFC 9000 §16 (big-endian, with
  the 2-bit length tag in the MSBs of the first byte).
- The first byte of the encoding satisfies:
  `varint_parse_len(first_byte) = varint_len(v)`.

### `get_varint()` (success case)
- The buffer offset advances by exactly `varint_parse_len(first_byte)`
  bytes.
- The returned value is the integer encoded in those bytes per RFC 9000.
- The returned value is ≤ `MAX_VAR_INT`.

## Key Invariants

### Round-Trip Correctness
For all `v ≤ MAX_VAR_INT` and any buffer with sufficient capacity:

```
decode(encode(v)) = v
```

That is: encoding `v` into a fresh buffer and then decoding from that
buffer at the same offset yields `v`.

This is the most important property: without it, every QUIC packet that
contains integers (which is all of them) could be silently corrupted.

### Length Consistency
For all `v ≤ MAX_VAR_INT`:

```
varint_parse_len(first_byte_of(encode(v))) = varint_len(v)
```

The parser can always determine the correct length from the first byte
alone, which is the key self-delimiting property of the encoding.

### Offset Advance
After a successful `put_varint(v)`, the offset increases by exactly
`varint_len(v)`. After a successful `get_varint()`, the offset increases
by exactly `varint_parse_len(first_byte)`.

## Edge Cases

1. **Value 0**: `varint_len(0) = 1`. Encoded as a single zero byte.
   Decoded as 0.

2. **Boundary values**:
   - `varint_len(63) = 1`, `varint_len(64) = 2`
   - `varint_len(16383) = 2`, `varint_len(16384) = 4`
   - `varint_len(1_073_741_823) = 4`, `varint_len(1_073_741_824) = 8`
   - `varint_len(MAX_VAR_INT) = 8`

3. **Value just above MAX_VAR_INT** (`= 2^62`): `varint_len` calls
   `unreachable!()`. `put_varint` panics (via `put_varint_with_len` with
   "value is too large for varint"). This is a programming error, not a
   network error.

4. **Buffer too short**: Both `put_varint` and `get_varint` return
   `Err(BufferTooShortError)` without modifying any state. The offset is
   NOT advanced on error.

5. **`put_varint_with_len(v, len)` with non-minimal `len`**: The API
   allows encoding a small value in a larger encoding (e.g., encoding 37
   in 2 bytes instead of 1). The decoded value is still 37. The round-trip
   property holds when the same `len` is used. The first byte's 2-bit tag
   still encodes `len` correctly.

## Inferred Intent

The implementation closely follows RFC 9000 §16. The key design choices:

1. **Self-delimiting**: the length is encoded in the first byte, so parsers
   never need out-of-band length information.
2. **Monotone encoding**: larger values use more bytes, so the 2-bit tag
   is a ceiling on what can be stored, not a variable-length code.
3. **Big-endian**: consistent with QUIC/TLS wire formats.
4. **Mask on decode**: the 2-bit tag is stripped by masking (`& 0x3fff`,
   etc.) during decoding, not by subtraction — this is important for the
   Lean model to match exactly.

## Open Questions

1. **Non-minimal encodings**: RFC 9000 §16 states that encoders should
   use the minimum necessary length, but does not explicitly say decoders
   must *reject* non-minimal encodings. The implementation accepts them
   silently. Is this intentional? The `put_varint_with_len` API explicitly
   supports non-minimal encodings.

2. **Value above MAX_VAR_INT**: the panic in `put_varint_with_len` (via
   the `_ => panic!()` arm) is a programmer error guard, but `put_varint`
   calls `varint_len` which calls `unreachable!()` for `v > MAX_VAR_INT`.
   Should there be a checked variant returning an error instead? This is an
   open design question.

3. **Byte order in `put_u` macros**: the implementation uses `ptr::copy_nonoverlapping`
   with `to_be` (big-endian). The Lean model will abstract this to a
   pure integer-to-bytes mapping; confirming this abstraction is sound
   would require a correspondence review.
