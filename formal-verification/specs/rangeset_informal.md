## Purpose

`RangeSet` is a sorted, non-overlapping collection of half-open `u64`
ranges `[start, end)`. It is used throughout the QUIC implementation to
track which packet numbers, stream bytes, or acknowledgement ranges have
been received or sent.

Two representations are maintained transparently:
- **`InlineRangeSet`**: backed by a `SmallVec` with capacity
  `MAX_INLINE_CAPACITY = 4`. No heap allocation when ≤ 4 ranges are
  stored.
- **`BTreeRangeSet`**: backed by a `BTreeMap<u64, u64>` for an
  arbitrary number of ranges.

The `fixup()` method migrates between representations automatically:
- Inline → BTree when `inner.len() == MAX_INLINE_CAPACITY = 4`
  (i.e., on insertion that would exceed the inline buffer).
- BTree → Inline when `inner.len() <= MIN_TO_INLINE = 2`.

A **capacity** parameter controls the maximum number of ranges stored.
When the limit is reached, the **smallest** (first in sort order) range
is evicted before inserting the new one.

---

## Key Invariant: Sorted Non-Overlapping Ranges

At all times, the stored ranges `r_0, r_1, ..., r_{n-1}` satisfy:

1. **Sorted**: `r_i.end ≤ r_{i+1}.start` for all `i`.
2. **Non-overlapping**: `r_i.end ≤ r_{i+1}.start` (same condition, since
   ranges are half-open).
3. **Non-adjacent** (gaps are preserved — adjacent ranges are NOT merged
   unless a new insertion explicitly bridges them): after every `insert`,
   if two ranges were adjacent before (e.g., `a..b` and `b..c`), they
   remain as two ranges unless the new insertion covers the gap.

   **Note**: The implementation merges overlapping ranges but does NOT
   pro-actively merge adjacent non-overlapping ranges. Whether adjacent
   ranges `a..b` and `b..c` are merged depends on whether the new
   insertion includes both `b-1` and `b`.
4. **Non-empty ranges**: `r_i.start < r_i.end` for all `i` (each stored
   range represents at least one value).

## Operations

### `RangeSet::new(capacity: usize)` → `RangeSet`

Creates an empty `RangeSet` with the given capacity.

**Postconditions**:
- `len() = 0`
- `iter()` yields no ranges.

---

### `insert(item: Range<u64>)`

Inserts `[item.start, item.end)` into the set. Overlapping and
touching existing ranges are merged.

**Preconditions**:
- `item.start < item.end` (non-empty range). If `item` is empty
  (`start == end`), the behaviour is unspecified (likely a no-op for
  the Inline variant; potentially incorrect for BTree).

**Postconditions** — let `before` be the abstract value-set before and
`after` be after:

1. **Set union**: `after = before ∪ [item.start, item.end)`.
   The abstract set of values covered is exactly the union.
2. **Invariant preserved**: `after` satisfies the sorted
   non-overlapping invariant.
3. **Capacity enforcement**: if `len() == capacity` before inserting
   and the insertion would increase `len()`, the smallest range is
   first removed. Specifically:
   - The range `r_0` with the smallest `start` is removed.
   - The new range is then inserted into the remaining set.
   - After this removal, the capacity constraint `len() ≤ capacity`
     holds.

**Important edge cases**:
- If `item` is entirely contained in an existing range, `len()` does
  not change and the abstract set does not change.
- If `item` spans multiple existing ranges (bridging their gaps), all
  spanned ranges are merged into one; `len()` decreases.
- If `item` extends an existing range from either side, the existing
  range is enlarged; `len()` does not change.
- If `item` is disjoint from all existing ranges, a new range is added;
  `len()` increases by 1 (unless capacity eviction occurs).
- Inserting an empty range (`item.start == item.end = 0..0`) causes
  undefined behaviour: `put_varint` is not called. The `push_item(v)`
  convenience method inserts `v..v+1` (always non-empty).

---

### `remove_until(largest: u64)`

Removes all values ≤ `largest` from the set. Equivalently: discards
any range `r` where `r.end ≤ largest + 1` (i.e., `r` is entirely
covered by `[0, largest]`); and trims the first range if it partially
overlaps `[0, largest]`.

**Postconditions**:
1. **Set difference**: `after = before \ [0, largest]` (as a set of
   `u64` values).
2. **No value ≤ `largest` remains**: `first() > largest` (or
   `first() = None` if the set is now empty).
3. **Values > `largest` are preserved**: if `v ∈ before` and
   `v > largest`, then `v ∈ after`.
4. **Invariant preserved**: `after` satisfies the sorted
   non-overlapping invariant.

---

### `push_item(item: u64)`

Convenience: equivalent to `insert(item..item+1)`.

---

### `iter()` → `impl Iterator<Item = Range<u64>>`

Returns an iterator over the stored ranges in ascending order of `start`.

**Postconditions**:
- Yields each stored range `r_i` exactly once.
- Ranges are in strictly ascending order: `r_i.end ≤ r_{i+1}.start`.
- Each yielded range `r` satisfies `r.start < r.end`.

---

### `len()` → `usize`

Returns the number of non-overlapping ranges stored.

---

### `first()` / `last()` → `Option<u64>`

- `first()`: the smallest value covered by any range (`r_0.start`), or
  `None` if empty.
- `last()`: the largest value covered by any range (`r_{n-1}.end - 1`,
  since ranges are half-open), or `None` if empty.

## Structural Invariants for Formal Verification

The following properties are the most valuable FV targets:

### I1 — Sorted order
```
∀ i < n-1,  r_i.end ≤ r_{i+1}.start
```

### I2 — Non-empty ranges
```
∀ i < n,  r_i.start < r_i.end
```

### I3 — Insert is a set union (correctness)
```
covers(insert(S, r)) = covers(S) ∪ [r.start, r.end)
```
where `covers(S) = ∪ { [r.start, r.end) | r ∈ S }`.

### I4 — remove_until is a set difference
```
covers(remove_until(S, v)) = covers(S) \ [0, v]
```

### I5 — Representation equivalence
Both `InlineRangeSet` and `BTreeRangeSet` maintain the same invariants
and implement the same operations; the `fixup()` transitions between
them preserve the logical value of the set.

## Examples (from tests)

| Operation | Result (`flatten()`) |
|-----------|----------------------|
| `insert(4..7)` | `[4, 5, 6]` |
| `insert(9..12)` | `[4, 5, 6, 9, 10, 11]` |
| `insert(5..7)` (overlapping) | `[4, 5, 6, 9, 10, 11]` (no change) |
| `insert(6..10)` (bridging) | `[4, 5, 6, 7, 8, 9, 10, 11]` after full bridge |
| `remove_until(4)` on `[3..6, 9..11]` | `[5, 9, 10]` |
| `remove_until(10)` on `[3..6, 9..11]` | `[]` |

Capacity test (capacity=3):
- After inserting `10..11`, `13..14`, `3..6` (3 ranges), the set has
  `{3..6, 10..11, 13..14}`.
- Inserting `16..20` (would create 4 ranges) evicts the smallest (`3..6`),
  leaving `{10..11, 13..14, 16..20}`.

## Edge Cases and Open Questions

1. **Empty range insertion** (`item.start == item.end`): the `InlineRangeSet`
   and `BTreeRangeSet` implementations differ slightly. The `BTreeRangeSet`
   would insert a zero-length range (which violates I2), while
   `InlineRangeSet` might produce unexpected behaviour. **Open question**:
   should `insert` check `item.start < item.end` and return early if not?

2. **Capacity vs. existing range merging**: when `len() == capacity` and a
   new insertion would merge with an existing range (not increasing `len()`),
   the capacity eviction is NOT triggered in the `BTreeRangeSet` (eviction
   only happens when `self.inner.len() >= self.capacity` after the point of
   insertion). **Open question**: is this the intended behaviour?

3. **u64 overflow in `push_item`**: `push_item(u64::MAX)` calls
   `insert(u64::MAX..u64::MAX + 1)` which would overflow. This is likely
   unreachable in practice since QUIC packet numbers are bounded.

4. **`remove_until` semantics for values beyond the set**: `remove_until(v)`
   where `v ≥ last()` correctly empties the set (confirmed by test).

5. **Iteration order**: the Inline variant uses a `SmallVec` (insertion is
   at the correct sorted position), and the BTree variant uses a `BTreeMap`
   (naturally sorted by key). Both iterate in ascending order.

## Proof Tractability Notes

- **I1 (sorted order)** after `insert`: provable by structural induction.
  The merge step makes the induction case non-trivial (the merged range
  is correctly bounded above and below). This is the most interesting
  correctness property.
- **I3 (insert = set union)**: this is the highest-value property. Lean
  can model `RangeSet` as `List (Nat × Nat)` (sorted list of intervals)
  and use set-theoretic lemmas. Requires careful treatment of the merge
  step. **Approximation needed**: the capacity-eviction side-effect means
  this property holds only when `len() < capacity` before insertion.
- **I4 (remove_until = set difference)**: more tractable, as `remove_until`
  only modifies the front of the sorted list.
- **Representation equivalence (I5)**: probably most amenable to Lean
  proofs, since both representations are isomorphic list/map structures.

The `BTreeRangeSet.insert` is the primary complexity source; the
`InlineRangeSet.insert` has a more explicit loop that is easier to reason
about inductively.
